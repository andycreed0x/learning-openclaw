From zero to a secure, production-ready deployment

Welcome to the final session of Learning OpenClaw. In this session we bring everything together: how to install OpenClaw, how to configure it, and how to secure it for production use. By the end, you will be ready to deploy your own personal AI assistant.

Installation Methods

Three ways to get started

npm (recommended)
npm install -g openclaw
Fastest way to get started. Requires Node.js.
From Source
git clone https://github.com/openclaw/openclaw.git
cd openclaw
npm install
npm run build
For contributors and custom builds.
Docker
docker pull openclaw/openclaw
Isolated, reproducible deployments.
There are three ways to install OpenClaw. The recommended method is via npm, which gives you the global openclaw command. If you want to contribute or customize the source, you can clone the repository and build from source. And for isolated, reproducible deployments, Docker is available. All three methods get you to the same result.

Prerequisites

What you need before installing

Node.js 20+
OpenClaw requires Node.js version 20 or higher. Download from nodejs.org.
npm 9+
Comes bundled with Node.js. Verify with npm --version.
At least one AI API key
You need a key from one of these providers:
  • OpenAI (sk-...)
  • Anthropic (sk-ant-...)
  • Google Gemini
  • Ollama (local — no key needed)
Quick check: node --version → v20+   |   npm --version → 9+
Before installing, make sure you have Node.js version 20 or higher and npm version 9 or higher. You will also need at least one AI provider API key. If you want to run completely offline, Ollama is an option that does not require an API key since it runs models locally on your machine.

Configuration File

openclaw.json5 — the heart of your setup

// openclaw.json5
{
  // Main configuration
  gateway: {
    port: 18789,
    host: "0.0.0.0"
  },

  agents: {
    default: {
      model: "gpt-4o",
      provider: "openai"
    }
  },

  channels: {
    web: { type: "web", port: 3000 }
  }
}
JSON5 format: Comments, trailing commas, unquoted keys — human-friendly configuration.
The configuration file uses JSON5 format, which is a superset of JSON that allows comments, trailing commas, and unquoted keys. The three main sections are gateway (network settings), agents (AI model configuration), and channels (messaging interfaces). This is the minimal configuration needed to get started.

Config Features

Powerful configuration capabilities

JSON5 Format
Comments allowed! Trailing commas, unquoted keys. Much more human-friendly than plain JSON.
Hot Reload on Save
Edit the config file and changes apply immediately. No need to restart the gateway.
$include for Splitting Files
Split your config into multiple files: 
agents: { $include: "./agents.json5" }
Environment Variable Substitution
Reference env vars directly: 
apiKey: "$OPENAI_API_KEY"
Never hardcode secrets in your config.
OpenClaw configuration has several powerful features. JSON5 means you can add comments to document your config. Hot reload means changes are picked up automatically without restarting. The $include directive lets you split large configs into multiple files. And environment variable substitution with the dollar sign prefix means you never need to hardcode API keys or secrets in your configuration file.

Security Overview

Multiple layers of protection

graph TB subgraph "Security Layers" DM["DM Policies
Who can talk?"] SB["Sandbox
Code isolation"] PI["Prompt Injection
Awareness"] AC["Access Control
Tool permissions"] end DM --> SB --> PI --> AC
Security in OpenClaw is defense in depth — multiple independent layers work together to protect your system, your data, and your users.
OpenClaw takes a defense-in-depth approach to security. There are four main layers. DM Policies control who can interact with your agent. The Sandbox isolates tool execution. Prompt Injection awareness helps prevent manipulation attacks. And Access Control restricts which tools each agent can use. These layers work independently, so even if one is bypassed, the others continue to protect your system.

DM Policies Recap

Control who can talk to your agent

Policy Behavior Best For
pairing New users must send a pairing code Personal use, high security
allowlist Only pre-approved users can chat Small teams, known users
open Anyone can chat with the agent Public bots, demos
disabled No DM conversations allowed Group-only bots
Recommendation: Use pairing mode for personal assistants. It requires a one-time pairing code, ensuring only you can access the agent.
DM Policies are your first line of defense. Pairing mode requires users to enter a code before they can start chatting, making it ideal for personal use. Allowlist mode lets you predefine which users can access the agent. Open mode allows anyone to chat, which is useful for public-facing bots. And disabled mode prevents DM conversations entirely, restricting the agent to group chats only.

Prompt Injection Awareness

Protecting against manipulation attacks

What is prompt injection?
An attacker crafts input that manipulates the agent into ignoring its instructions and performing unintended actions.
Built-in protections
  • System prompt isolation
  • Tool call validation
  • Output filtering
  • Sandbox isolation for exec tools
User awareness recommendations
  • Restrict tool permissions to the minimum needed
  • Enable sandbox mode in production
  • Review agent logs regularly
  • Use allowlist DM policies when possible
  • Be cautious with tools that access external content
Prompt injection is a serious concern for any AI assistant that can take actions. An attacker could craft input designed to make the agent bypass its instructions and execute harmful commands. OpenClaw includes built-in protections like system prompt isolation and tool call validation. However, no protection is perfect, so it is important to follow best practices: restrict tool permissions, enable sandboxing, review logs, and use strict DM policies.

Deployment Tips

Running OpenClaw in production

Process Manager
Use systemd or pm2 to keep OpenClaw running. 
pm2 start openclaw --name oc
pm2 save
pm2 startup
Reverse Proxy
Put nginx or Caddy in front of OpenClaw for SSL termination, load balancing, and static file serving.
Enable HTTPS
Always use HTTPS in production. Use Let's Encrypt for free certificates. Never expose the gateway over plain HTTP.
Regular Backups
Back up your workspace/ directory regularly. It contains conversations, skills, and agent state. 
tar -czf backup.tar.gz workspace/
For production deployments, use a process manager like pm2 or systemd to ensure OpenClaw stays running and restarts on crashes. Put a reverse proxy like nginx in front for HTTPS, which you should always enable in production. And back up your workspace directory regularly, as it contains all your conversations, custom skills, and agent state data.
