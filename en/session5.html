Extending your agent with real-world capabilities

Welcome to Session 5. In this session we explore how OpenClaw agents go beyond conversation. Tools let agents interact with the real world, Skills package reusable functionality, and the Sandbox keeps everything safe.

What are Tools?

Tools extend what agents can do beyond conversation

Without Tools
Agents can only generate text responses. They are limited to what the AI model knows.
With Tools
Agents can browse the web, read files, execute code, send messages, schedule tasks, and more.
Tools allow agents to interact with the real world — reading data, performing actions, and returning results back to the conversation.
Think of tools as the hands and eyes of your AI agent. Without tools, an agent can only talk. With tools, it can browse websites, read and write files, run shell commands, schedule reminders, and much more. Tools are what make OpenClaw agents truly useful in day-to-day tasks.

Tool Categories

Category Description Examples
Browser Web browsing & scraping Navigate, screenshot, extract
Canvas Visual content creation Draw, generate images
Node System operations File read/write, HTTP requests
Cron Scheduled tasks Reminders, periodic checks
Webhooks HTTP endpoints Receive external triggers
Exec Code execution Run scripts, commands
OpenClaw organizes tools into six categories. Browser tools let agents navigate the web. Canvas tools handle visual content. Node tools perform system-level operations. Cron tools schedule recurring tasks. Webhooks allow external systems to trigger the agent. And Exec tools run shell commands and scripts. Each category can be enabled or disabled independently for each agent.

Tool Execution Flow

How the agent decides to use a tool

graph TB A["Agent receives request"] --> D{"Needs tool?"} D -->|No| R["Direct response"] D -->|Yes| T["Select tool"] T --> S{"Sandbox enabled?"} S -->|Yes| SB["Run in sandbox"] S -->|No| EX["Run directly"] SB --> RES["Tool result"] EX --> RES RES --> A2["Agent processes result"] A2 --> RESP["Final response"]
When an agent receives a request, it first decides whether a tool is needed. If not, it responds directly. If a tool is needed, the agent selects the appropriate one. The system then checks whether sandboxing is enabled. If so, the tool runs in an isolated environment. The result is returned to the agent, which processes it and formulates the final response to the user.

Headless Chromium via Puppeteer

  • Navigate to any URL

  • Extract page content
  • Wait for dynamic content
  • Handle multiple tabs

         and sunny in NYC"
The browser tool runs a real headless browser — it can interact with JavaScript-heavy sites, SPAs, and pages that require authentication.
The Browser tool is one of the most powerful tools in OpenClaw. It runs a headless Chromium browser using Puppeteer. This means the agent can navigate to any website, take screenshots, click buttons, fill out forms, and extract content. It handles JavaScript-rendered pages, single-page applications, and even authenticated sessions. A common use case is having the agent check a website and summarize the information.

Exec Tool

Execute shell commands safely

  • Run shell commands
  • Configurable allowed commands
  • Sandbox isolation available
  • Timeout protection
Security tip: Always restrict allowed commands in production to prevent unintended system access.

Example configuration:

// openclaw.json5
{
  tools: {
    exec: {
      enabled: true,
      allowedCommands: [
        "ls", "cat", "grep",
        "curl", "ping",
        "node", "python3"
      ],
      timeout: 30000,
      sandbox: true
    }
  }
}
The Exec tool lets agents run shell commands on the host system. This is extremely powerful but must be configured carefully. You define a whitelist of allowed commands, set timeouts to prevent runaway processes, and optionally enable sandbox isolation. In production, you should always restrict the allowed commands to only what the agent genuinely needs.

What are Skills?

Reusable prompt + tool packages

Think of Skills as "apps"
A Skill bundles a system prompt, tool configuration, and behavior into a single reusable package that any agent can use.
Composable
Agents can use multiple Skills simultaneously. Skills can be combined to create complex workflows.

Example skill definition:

// skills/web-search.json5
{
  name: "web-search",
  description: "Search the web",
  prompt: "You can search the web
    using the browser tool.
    Always cite your sources.",
  tools: ["browser"],
  triggers: [
    "search for",
    "look up",
    "find online"
  ]
}
Skills are one of the most powerful concepts in OpenClaw. A Skill is essentially an app for your agent. It bundles together a system prompt that defines behavior, tool configurations, and optional trigger phrases. Skills are composable, meaning an agent can use multiple skills at the same time. You can think of skills as modular capabilities that you plug into your agent.

Skills come from three sources. Bundled skills ship with OpenClaw and cover common use cases like summarization, translation, and web search. Workspace skills are custom skills you create in your workspace's /skills directory. And ClawHub is the community marketplace where you can browse, install, and share skills with other OpenClaw users. You can mix and match skills from all three sources.

Sandbox Modes

Control how tool execution is isolated

Mode Behavior Use Case
off No sandbox — tools run directly on the host Development, trusted environments
non-main Sandbox for non-main agents only Main agent trusted, sub-agents isolated
all Sandbox everything — all tool execution isolated Production, multi-user, untrusted input
Security implication: With sandbox off, a prompt injection attack could lead to arbitrary code execution on your system. Always use all in production or when processing untrusted input.
Sandbox mode controls how tool execution is isolated from the host system. In "off" mode, tools run directly on the host with no isolation. This is fine for development but dangerous in production. The "non-main" mode sandboxes sub-agents while trusting the main agent. The "all" mode sandboxes everything, which is the recommended setting for production environments. Remember, without sandboxing, a successful prompt injection could lead to arbitrary code execution.

Sandbox Architecture

Isolated execution with isolated-vm

graph TB subgraph "Main Process" AG["Agent"] end subgraph "Sandbox (isolated-vm)" EX["Code Execution"] FS["Limited FS Access"] NET["Network Restrictions"] end AG -->|"sandboxed call"| EX EX -->|"result"| AG
What the sandbox provides:
Memory isolation, CPU time limits, restricted filesystem access, network restrictions, no access to main process
What the sandbox prevents:
Reading arbitrary files, making unauthorized network calls, accessing environment variables, spawning child processes
The sandbox uses isolated-vm to create a completely separate execution environment. Code running inside the sandbox cannot access the main process memory, cannot read arbitrary files, cannot make unauthorized network calls, and cannot spawn child processes. The sandbox enforces memory limits, CPU time limits, and restricted filesystem access. Results are passed back to the agent through a controlled channel.

Key Takeaways

Tools give agents real-world capabilities
From browsing the web to executing code, tools transform conversational agents into powerful assistants that can take action.
Skills are reusable packages
Bundle prompts, tools, and triggers into shareable, composable "apps" for your agent. Use bundled skills, create your own, or install from ClawHub.
The Sandbox protects your system
Always enable sandboxing in production. It isolates tool execution, preventing unauthorized access to your host system.
Let us recap the three key takeaways from this session. First, tools are what give agents the ability to interact with the real world. Second, skills let you package and reuse tool-plus-prompt combinations as modular apps. Third, the sandbox is your safety net, isolating tool execution so that even if something goes wrong, your system remains protected.
